# /orchestrator-new — Greenfield Project Bootstrapper

**YOU ARE A PROJECT ARCHITECT.** You take a project idea from the user, research the
optimal architecture via Perplexity, generate a comprehensive BLUEPRINT.md with all phases
and sub-phases, then route to `/orchestrator` or `/orchestrator-multi` for execution.
You do NOT implement code yourself — you design and delegate.

**Architecture**: Intake -> Research (2-pass) -> Blueprint -> Route -> Handoff. $0 research
cost via Perplexity browser automation.

**CRITICAL: This command is for GREENFIELD PROJECTS ONLY. If the target directory contains
existing source code, redirect the user — do NOT proceed.**

## Input

`$ARGUMENTS` = `<project-prompt> [--dir <path>] [--stack <tech>] [--multi] [--single]`

| Flag | Description | Default |
|------|-------------|---------|
| `--dir <path>` | Target directory for the new project | `./<project-slug>` in cwd |
| `--stack <tech>` | Force specific tech stack (e.g., `nextjs`, `fastapi`, `react+express`) | auto (research decides) |
| `--multi` | Force multi-agent routing | auto |
| `--single` | Force single-agent routing | auto |

**Parse rules:**
1. If `$ARGUMENTS` is empty → STOP: "Provide a project description. Example: `/orchestrator-new A real-time chat app with rooms, auth, and message history`"
2. Extract `--dir`, `--stack`, `--multi`, `--single` flags if present
3. Remaining text = the project prompt

## Phase A: PROJECT INTAKE & PREFLIGHT

### Step 1: Parse & Validate the Project Prompt

1. Extract the project description from `$ARGUMENTS`
2. Identify any stated constraints: tech preferences, deployment target, user count, integrations
3. If the prompt is too vague (< 10 words, no clear deliverable), ask ONE clarifying question via `AskUserQuestion`:
   - "What type of application?" with options: Web App, API/Backend, CLI Tool, Mobile App
   - Do NOT ask more than one question — proceed with reasonable defaults for anything else

### Step 2: Check for Existing Code

Scan the target directory (from `--dir` or inferred from project slug):

```bash
# If directory exists, check contents
ls <target-dir> 2>/dev/null | head -20
```

**Decision tree:**
- Directory does not exist → proceed (will create in Step 3)
- Directory exists but empty / only `.git`, `.gitignore`, `README.md` → proceed (use as-is)
- Directory exists with source code (`.py`, `.ts`, `.js`, `.go`, `.rs`, etc.) → **STOP and present options:**

```
This directory contains existing source code. /orchestrator-new is for greenfield projects.

Options:
1. Use /orchestrator instead (manages existing codebase) — RECOMMENDED
2. Specify a different empty directory with --dir <path>
3. Override with --force (dangerous — ignores existing files)
```

Use `AskUserQuestion` to let the user choose. If option 1, invoke `/orchestrator` via `Skill` tool with the project path + prompt and STOP.

### Step 3: Create Project Directory & Git Repo

```bash
mkdir -p <target-dir>
cd <target-dir>
git init
```

Generate `.gitignore` based on detected or stated stack:
- If `--stack` provided → use stack-appropriate ignores
- If no stack yet → write a universal `.gitignore` (`.env`, `node_modules/`, `__pycache__/`, `.venv/`, `dist/`, `build/`, `.DS_Store`, `*.log`)

```bash
git add .gitignore && git commit -m "chore: initial commit with .gitignore"
```

### Step 4: Persist the Original Prompt

Write `PROJECT_PROMPT.md` to the project root — preserves the user's original intent
verbatim so downstream commands never lose context:

```markdown
# Project Prompt

## Original Request
{exact user prompt from $ARGUMENTS}

## Stated Constraints
- Stack: {--stack value or "auto"}
- Routing: {--multi or --single or "auto"}
- Directory: {target-dir}

## Generated By
/orchestrator-new — {ISO 8601 timestamp}
```

```bash
git add PROJECT_PROMPT.md && git commit -m "docs: persist original project prompt"
```

### Step 5: Create Scaffold Directories

```bash
mkdir -p src docs .workflow
```

## Phase B: ARCHITECTURE RESEARCH (2-Pass via Perplexity)

**This is the critical intelligence-gathering phase. Two mandatory Perplexity passes —
one for design, one for adversarial critique.**

### Step 1: Research Pass 1 — Stack & Structure

Build the research query. If `--stack` was provided, reframe the stack section as validation
rather than selection ("Given that I'm using {stack}, what is the optimal structure?").

Invoke `/research-perplexity` via `Skill` tool with this query (note: `/research-perplexity`
will add its own workflow on top — let it run its full workflow, but the core query is):

```
[ENVIRONMENT CONTEXT — READ FIRST]
This project is being developed using Claude Code, Anthropic's official CLI tool for Claude
(claude.ai/claude-code). The developer uses a Claude Max subscription and works entirely in
the terminal via the `claude` CLI command. Claude Code is an agentic coding assistant that
reads/writes files, runs terminal commands, searches codebases, and executes multi-step
development tasks autonomously. All code generation, refactoring, debugging, and project
management happens through Claude Code's conversation interface — there is no IDE or GUI
involved. Responses should account for this workflow: recommend CLI-compatible tools,
terminal-based solutions, and approaches that work well with an AI coding agent operating
in a command-line environment.
[END ENVIRONMENT CONTEXT]

I am bootstrapping a brand new greenfield project. I need a comprehensive architectural
recommendation. The project does not yet exist — I'm starting from zero.

PROJECT BRIEF: {user's project prompt}

STATED CONSTRAINTS: {--stack value if any, deployment target if mentioned, user count if mentioned}

Provide a structured architectural recommendation covering ALL of the following:

1. TECH STACK SELECTION
   - Primary language/runtime and why (fit for this specific problem, not just popularity)
   - Framework recommendation with current community health assessment
   - Database type and specific choice with rationale
   - Authentication approach (if applicable)
   - Key libraries/packages for core functionality
   {If --stack provided: "The user has chosen {stack}. Validate this choice and identify
   any concerns, but do not suggest alternatives unless there are critical issues."}

2. DIRECTORY STRUCTURE
   - Exact top-level layout as a tree diagram
   - Module boundary reasoning — why these divisions?
   - Monorepo vs single-repo decision with rationale

3. BUILD TOOLCHAIN
   - Package manager recommendation
   - Build/bundle tool
   - Test runner and testing strategy
   - Linter/formatter configuration
   - Minimum viable CI pipeline structure

4. PHASED IMPLEMENTATION ORDER
   - Phase 1: What must exist before anything else (the foundation)
   - Phases 2-N: Ordered by dependency, NOT arbitrary
   - For each phase: title, description, specific files/directories in scope, estimated
     effort (S/M/L/XL), can it run in parallel with other phases? (YES/NO + explanation)
   - Maximum 10 phases — combine small related work into single phases
   - Each phase must have clear acceptance criteria (testable, not vague)

5. CRITICAL EARLY DECISIONS
   - Decisions that are expensive to reverse after Phase 1
   - What to stub vs implement fully in Phase 1
   - Configuration/environment variable strategy
   - API design approach (if applicable)

6. KNOWN PITFALLS
   - Common mistakes for this type of project
   - Security considerations unique to this domain
   - Performance traps to avoid early
   - Dependency choices that commonly cause regret
```

**IMPORTANT**: When `/research-perplexity` runs its own workflow (entering plan mode, doing
verification, etc.), let it complete fully. But extract only the Perplexity research results
— not the plan it generates. The plan structure from `/research-perplexity` is for its own
internal use; you need the raw architectural recommendations.

**Wait for Pass 1 to complete before proceeding to Pass 2.**

### Step 2: Research Pass 2 — Adversarial Critique

Run a second query:

```
[ENVIRONMENT CONTEXT — READ FIRST]
This project is being developed using Claude Code, Anthropic's official CLI tool for Claude
(claude.ai/claude-code). The developer uses a Claude Max subscription and works entirely in
the terminal via the `claude` CLI command. Claude Code is an agentic coding assistant that
reads/writes files, runs terminal commands, searches codebases, and executes multi-step
development tasks autonomously. All code generation, refactoring, debugging, and project
management happens through Claude Code's conversation interface — there is no IDE or GUI
involved. Responses should account for this workflow: recommend CLI-compatible tools,
terminal-based solutions, and approaches that work well with an AI coding agent operating
in a command-line environment.
[END ENVIRONMENT CONTEXT]

I am building: {user's project prompt}

A previous research pass recommended this architecture:
{200-word summary of Pass 1 findings: stack, structure, phase count, key decisions}

ADVERSARIAL CRITIQUE — Challenge this recommendation:

1. WEAKEST ASSUMPTIONS: What assumptions in this recommendation are most likely to be wrong?
2. ALTERNATIVE STACKS: Would a different stack outperform this for maintenance cost,
   developer velocity, or scaling? (Only suggest alternatives with clear advantages)
3. PHASE ORDERING ERRORS: Are any phases ordered wrong? Should anything be earlier/later?
4. MISSING WORK: What's entirely missing from the plan? (auth, error handling, monitoring,
   deployment, testing strategy, CI/CD)
5. SCOPE CREEP RISK: What in this plan is unnecessary for an MVP? What should be deferred?
6. CLAUDE CODE WORKFLOW FIT: Some "best practices" for large teams are complexity overhead
   for a solo AI-assisted developer. Which recommendations fall into this trap?
7. REVISED PHASE LIST: If you would change the phases, provide your revised version with
   the same structure (title, files, effort, parallelizable, acceptance criteria).
```

### Step 3: Persist Research Findings

Write both pass results to `docs/RESEARCH_FINDINGS.md`:

```markdown
# Research Findings — {Project Name}

**Generated**: {ISO 8601 timestamp}
**Research Scope**: STRATEGIC (architecture & blueprint level)
**Passes**: 2 (Stack & Structure + Adversarial Critique)

## Pass 1: Stack & Structure Recommendation
{full Pass 1 results}

## Pass 2: Adversarial Critique
{full Pass 2 results}

## Synthesized Decisions
{reconcile any conflicts between Pass 1 and Pass 2 — pick the stronger argument}
```

```bash
git add docs/RESEARCH_FINDINGS.md && git commit -m "docs: architecture research findings (2-pass)"
```

### Error Handling for Phase B

If either research pass fails:
1. Retry once
2. If retry fails: proceed with whatever results were obtained. If Pass 1 failed entirely,
   STOP — "Architecture research failed. Check Perplexity session (`/cache-perplexity-session`).
   Cannot generate blueprint without research."

### Step 2: Validate Blueprint Completeness

Before proceeding, verify every phase has ALL required fields:
- [ ] Title and effort estimate

If any phase is incomplete, fill in the gap from research findings. If research didn't
cover it, make a reasonable default and mark it: `<!-- INFERRED: not from research -->`

```bash
git add BLUEPRINT.md && git commit -m "docs: comprehensive implementation blueprint"
```

## Phase D: ROUTING DECISION

### Step 1: Compute Routing Score

Analyze the BLUEPRINT.md to compute a weighted routing score:

| Signal | Condition | Score |
|--------|-----------|-------|
| Independent parallel phases | 3+ phases marked `Parallelizable: YES` with non-overlapping file scopes | +1 |
| Low file scope overlap | Max overlap between any two parallel phases < 20% of their combined files | +1 |
| Large total effort | Sum of efforts = 5+ Medium phases or 3+ Large/XL phases | +1 |
| Distinct module directories | 3+ top-level modules (e.g., `api/`, `frontend/`, `workers/`, `shared/`) | +1 |
| Monorepo structure | Research recommended `packages/` or multi-package layout | +1 |
| User requested `--multi` | Flag present | +2 |
| User requested `--single` | Flag present | -3 |
| Strong sequential chain | >60% of phases form a strict A→B→C dependency chain | -1 |
| Infrastructure-only project | No app code — terraform, k8s, ansible, helm | -2 |

**Routing rule:**
- Score >= 3 → `/orchestrator-multi`
- Score < 3 → `/orchestrator`

### Step 2: Record Routing Decision

Update BLUEPRINT.md's `## Routing Analysis` section:

```markdown
## Routing Analysis
| Signal | Value | Score |
|--------|-------|-------|
| Independent parallel phases | {count} phases parallelizable | {+1 or 0} |
| File scope overlap | {max %}% | {+1 or 0} |
| Total effort | {sum} | {+1 or 0} |
| Module directories | {count} distinct | {+1 or 0} |
| Monorepo structure | {yes/no} | {+1 or 0} |
| User preference | {--multi/--single/none} | {score} |
| Sequential dependency chain | {%} of phases | {-1 or 0} |
| Infrastructure-only | {yes/no} | {-2 or 0} |
| **TOTAL** | | **{total}** |
| **VERDICT** | | **{SINGLE_AGENT or MULTI_AGENT}** |
| **Reason** | {1-line justification} | |
```

```bash
git add BLUEPRINT.md && git commit -m "docs: routing decision — {SINGLE_AGENT|MULTI_AGENT}"
```

### Step 3: Report to User

Present the routing decision before handoff:

```
Project: {name}
Stack: {primary stack}
Phases: {count} ({count} parallelizable)
Routing: {/orchestrator or /orchestrator-multi} — {1-line reason}

Research: docs/RESEARCH_FINDINGS.md

Proceeding to {orchestrator/orchestrator-multi}. Continue? [y/n]
```

Wait for user confirmation via `AskUserQuestion` before Phase E.

### Step 2: Invoke Downstream Command

**If routing = SINGLE_AGENT:**

Invoke `/orchestrator` via `Skill` tool with args: `<project-dir> "Implement all phases in BLUEPRINT.md. Read CLAUDE.md for genesis context, known decisions, and pitfalls. Start with Phase 1."`

The downstream `/orchestrator` will:
- Detect `## Project Genesis` in CLAUDE.md → skip Steps 2.5 (scaffold), Step 3.5 research
  runs but for tactical/phase-level guidance (not strategic/blueprint-level)
- Assess maturity as **Scaffold** tier (new project)
- Proceed to Phase B (launching loop_driver.py)

**If routing = MULTI_AGENT:**

Invoke `/orchestrator-multi` via `Skill` tool with args: `<project-dir> "Implement all phases in BLUEPRINT.md using parallel agents. Territory map is in BLUEPRINT.md dependency graph and file scope annotations. Read CLAUDE.md for genesis context."`

The downstream `/orchestrator-multi` will:
- Detect BLUEPRINT.md with phase annotations → use for territory splitting (Phase 0 Path A)
- Skip Appropriateness Gate (routing already decided by /orchestrator-new)
- Proceed to Phase A Step 2 (analyze work split from BLUEPRINT.md)

### Step 3: STOP

After invoking the downstream command, **STOP**. The downstream command takes over entirely.
Do not monitor, do not intervene, do not duplicate any downstream work.

## Edge Cases

| Scenario | Action |
|----------|--------|
| Existing code in target dir | Present options: /orchestrator, new dir, or --force (Step A.2) |
| Monorepo requested | Research query includes monorepo structure; routing score +1 |
| Exact stack specified (`--stack`) | Pass 1 validates instead of selecting; structure/phases still researched |
| Infra-only project (terraform, k8s, etc.) | Routing score -2; defaults to single-agent |
| Research Pass 1 fails | Retry once; if still fails, STOP (cannot generate blueprint without research) |
| Research Pass 2 fails | Proceed with Pass 1 only; note in RESEARCH_FINDINGS.md |
| User prompt too vague | Ask ONE clarifying question (app type); proceed with defaults for rest |
| `--multi` AND `--single` both set | Error: "Cannot specify both --multi and --single" |

## Integration Notes

### How Downstream Commands Detect /orchestrator-new Handoff

The `## Project Genesis` section in CLAUDE.md is the detection signal:

- **`/orchestrator`**: If CLAUDE.md contains `## Project Genesis`, skip Step 2.5 (scaffold
  generation) and use the existing CLAUDE.md phases. Step 3 (maturity assessment) auto-classifies
  as **Scaffold** tier. Step 3.5 (research) still runs but for tactical phase-level guidance,
  referencing `docs/RESEARCH_FINDINGS.md` for strategic context already gathered.

- **`/orchestrator-multi`**: If BLUEPRINT.md exists with `## Routing Analysis` showing
  `MULTI_AGENT` verdict, Phase 0 Auto-Discovery uses Path A (BLUEPRINT.md exists) and
  skips the Appropriateness Gate (already decided).

### Research Scope Layering

| Pass | Command | Scope | Question |
|------|---------|-------|----------|
| Pass 1 | /orchestrator-new | Strategic | "What should I build with?" |
| Pass 2 | /orchestrator-new | Adversarial | "What's wrong with Pass 1?" |
| Pass 3 | /orchestrator Step 3.5 | Tactical | "How do I implement Phase N specifically?" |

These are NOT redundant — each answers a fundamentally different question.
